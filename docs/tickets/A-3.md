## 2. Architectural Decisions Made in A-3

### 2.1 `chapters.py` lives in the library layer

Per A-2 §4.1, library logic must not depend on argparse. `chapters.py`
imports only `dataclasses`, `logging`, `pathlib`, and (lazily, inside
the path-based wrapper) `pymupdf`. No CLI knowledge anywhere. The
`--show-chapters` flag's plumbing lives in `preprocess.py`, the
argparse-facing seam.

This pattern (library module + thin orchestration in `preprocess.py`)
will be repeated by A-4 (heuristic chapter detection), A-5 (trim),
A-6 (cover), A-7 (pagination), and A-8 (.book writer). **Do not break
this pattern.**

### 2.2 Page numbers are 0-based at our boundary

PyMuPDF's `get_toc()` returns 1-based page numbers (verified during
A-3; see citation in `chapters.py` docstring). This project standardises
on **0-based page numbers internally**, because:

- Iterating `for page in doc` yields pages 0..N-1.
- Slicing/list indexing is universally 0-based in Python.
- A-7's pagination engine will work with page lists, where 1-based
  indexing would invite off-by-ones.

The conversion happens exactly once, in `detect_chapters_from_outline`,
at the boundary between PyMuPDF output and our domain. Downstream code
should never see a 1-based value from this module.

**User-facing log output displays 1-based pages** (in `preprocess.run`
under `--show-chapters`) so the numbers match what PDF readers show.
The stored `Chapter.start_page` value stays 0-based. This split is
intentional; don't normalise both sides to the same convention.

### 2.3 `Chapter` is frozen and hashable

`@dataclass(frozen=True)` so instances are hashable and immutable.
Useful for set-based diffing, equality in tests, and ruling out
accidental in-place mutation by later code.

Fields kept minimal — `title`, `start_page`, `level`. If A-4 / A-5
need additional fields (e.g. an `end_page` after trim is applied, or a
`source: Literal["outline", "heuristic"]` tag), add them with explicit
defaults so existing callers continue to construct `Chapter` positionally.

### 2.4 Unresolved outline entries are skipped, not raised

PyMuPDF emits `page == -1` for outline entries that don't resolve to a
concrete page (external URLs, broken named destinations, etc.).
`detect_chapters_from_outline` skips these and logs a single summary
warning. Rationale: the same defensive philosophy as A-2's empty-page
handling (§4.5 of A-2.md). A real-world PDF with a couple of broken
bookmarks should still produce a useful chapter list.

### 2.5 Outline hierarchy levels are preserved verbatim

We don't flatten the outline, and we don't filter to level-1 only by
default. Sub-chapters are first-class entries in the returned list.
`top_level_chapters()` is provided as a separate convenience for callers
that want only the top of the tree. The `.book` writer in A-8 can
decide which levels to emit; A-3 only collects the raw data.

### 2.6 `--show-chapters` is a flag on `preprocess`, not an `inspect` subcommand

Plan §3.1 envisions an `--inspect` dry-run mode. A-1.md §3.4 mapped that
to a sibling `inspect` subcommand. A-3 deliberately *does not* introduce
`inspect` yet — that subcommand is A-5's territory (trim dry-run is the
core inspect functionality).

Until then, `--show-chapters` on `preprocess` is the cheapest path to
making A-3's acceptance check user-visible. **When A-5 introduces
`inspect`, the `--show-chapters` flag here will likely be removed and
the chapter-list output folded into the `inspect` subcommand.** This is
a deliberate temporary surface; flag in any deprecation notice when
that migration happens.

### 2.7 Tests build their own PDFs via `set_toc`

`set_toc` and `get_toc` round-trip the simple format
`[level, title, page_1_indexed]` losslessly. This lets the test suite
run with zero external fixtures, which matters for CI and for any
future agent reproducing the test environment.

The round-trip is verified by PyMuPDF itself, not by our tests. If
PyMuPDF ever introduces silent normalisation between `set_toc` and
`get_toc` (e.g. clamping page numbers or rewriting whitespace), some
of our tests would falsely pass without exercising the production
parsing path. This is an acceptable risk for A-3 but worth knowing
about for any future change to the dependency.

## 3. PyMuPDF API Surface Used in A-3 (verified)

A-3 newly uses (all verified during the A-3 session against PyMuPDF
1.27.2.3, both via the official docs and by running the test suite):

- `pymupdf.open()` (no args) — create a new empty PDF in memory.
- `pymupdf.open(path)` — open an existing PDF.
- `doc.new_page()` — append a blank page (used by tests only).
- `doc.set_toc(toc)` — set the outline (used by tests only).
- `doc.get_toc(simple=True)` — read the outline as
  `[[level, title, page_1_indexed], ...]`.
- `doc.save(path)` — write the document to disk (used by tests only).
- `doc.close()` — explicit close.

The import name is `pymupdf` (the legacy `fitz` alias is not used
anywhere in this codebase).

## 6. Hand-Off Notes for A-4

A-4 is "Chapter detection heuristic fallback" — for PDFs without an
outline, detect chapter starts via regex on page text (e.g.
`^Chapter \d+`, `^Prologue`, `^Introduction`). Acceptance: works on at
least one PDF without bookmarks.

Things the next agent should know before starting A-4:

1. **Read `PROJECT_API.md` and `PROJECT_TREE.md` first.** As of A-3
   close they reflect the current state. Verify against the actual code
   if anything looks off; the docs are best-effort, the code is truth.

2. **Verify A-3 was applied.** Look for
   `src/gbc_reader_prep/chapters.py` and
   `tests/test_chapters.py`. If either is missing, A-3 was drafted but
   not applied; confirm with the user before extending.

3. **A-4 extends `chapters.py`, doesn't replace it.** Add a new public
   function `detect_chapters_heuristic(doc)` (or similar) alongside the
   existing outline-based function. **Keep the same `Chapter` dataclass**
   — heuristic detection should produce the same record type so
   downstream code (A-5 onward) doesn't care which source produced the
   chapter list.

4. **Suggested signature:**
   ```python
   def detect_chapters(doc) -> list[Chapter]:
       """Use the outline if present; otherwise fall back to heuristics."""
   ```
   Returns either the outline-based list or the heuristic list. A
   `source: Literal["outline", "heuristic", "none"]` annotation either
   in the `Chapter` record or in a separate return value is worth
   considering — would help A-5's dry-run report be clear about how
   the chapter list was obtained.

5. **Suggested regex patterns to start with**, all case-insensitive,
   anchored to the start of a page's first non-empty line:
   - `^chapter\s+\d+\b` (e.g. "Chapter 1", "CHAPTER 12")
   - `^chapter\s+(one|two|three|four|five|six|seven|eight|nine|ten|...)\b`
     (English ordinals; consider how far to go)
   - `^prologue\b`, `^epilogue\b`, `^introduction\b`, `^foreword\b`,
     `^preface\b`
   - Roman numerals: `^(part|chapter|book)\s+[ivxlcdm]+\b`
   These are starting points; do not over-engineer A-4. The plan calls
   for "works on at least one PDF without bookmarks" — that's a low bar
   intentionally. Don't try to handle every weird novel layout in A-4.

6. **Test fixture strategy.** PyMuPDF can build a PDF with arbitrary
   text content via `page.insert_text((x, y), "Chapter 1...")`. Use this
   to build heuristic-detection fixtures the same way A-3 used
   `set_toc` for outline fixtures. Verify the text-extraction round-
   trip works the way A-3 verified the TOC round-trip — by reading back
   what was written.

7. **Don't change `--show-chapters`'s wiring.** It already calls
   `detect_chapters_from_outline_path`. A-4 might want to switch it
   to a unified `detect_chapters` (outline-then-heuristic) entry point.
   If so, update the flag's `help=` text to reflect the new behaviour.

8. **PyMuPDF text extraction** is `page.get_text()` (default mode
   `"text"`) per A-2. Use the same call for consistency. Heuristic
   detection only needs the **first line** of each page in most cases;
   `page.get_text("text").splitlines()[0]` is enough.
