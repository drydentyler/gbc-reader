## 2. Architecture Established in A-2

A-2 introduces a small but important pattern that should propagate to all
later subcommands:

```
┌────────────────────────────────────────────────────────┐
│ cli.py                                                 │
│   build_parser() — top-level args + subparsers group   │
│   main()         — dispatches via args.func            │
└──────┬─────────────────────────────────────────────────┘
       │ registers
       ▼
┌────────────────────────────────────────────────────────┐
│ preprocess.py  (subcommand layer — argparse-coupled)   │
│   SUBCOMMAND = "preprocess"                            │
│   add_subparser(subparsers) — registers + sets func    │
│   run(args)   — handler, returns int exit code         │
└──────┬─────────────────────────────────────────────────┘
       │ calls
       ▼
┌────────────────────────────────────────────────────────┐
│ extract.py  (library layer — framework-agnostic)       │
│   extract_text(pdf_path, out_path) -> int              │
└────────────────────────────────────────────────────────┘
```

**Why the split:** A-3 onward will add chapter detection, trimming,
pagination, and a `.book` writer. Each is a meaningful library unit. If
all the logic lived in `preprocess.py` it would be a god module by A-8.
Instead, `preprocess.py` is the argparse-facing seam that orchestrates
calls into library modules. Library modules know nothing about argparse.

**Implications for future tickets:**
- A-3 should create `chapters.py` (or similar) as a new library module
  rather than adding chapter logic to `preprocess.py` or `extract.py`.
- Future subcommands (`inspect` is mentioned in plan §3.1 and A-1.md §3.4)
  should follow the `preprocess.py` shape exactly: `SUBCOMMAND` constant,
  `add_subparser(subparsers)`, `run(args)`, `parser.set_defaults(func=run)`.

## 4. Architectural Decisions Made in A-2

### 4.1 `extract.py` is intentionally framework-agnostic

`extract_text(pdf_path, out_path) -> int` does one thing: read text from a
PDF, write it to a file with page separators. No argparse, no CLI
concerns, no knowledge of the `preprocess` subcommand. This lets
future modules (chapter detection, pagination, `.book` writer) call it
directly without going through the CLI layer.

### 4.2 Subcommand name is `preprocess`, not `extract`

A-1.md §8.5 explicitly directs: "add the `preprocess` subcommand skeleton
in A-2 (even if it only writes `.txt` so far)." Naming was already decided
in A-1. **Do not rename it in later tickets** — A-3 through A-8 evolve
the implementation underneath the same `preprocess` subcommand. By A-8
the same command produces a `.book` file; by then the A-2 stub behavior
will be replaced.

If a separate inspection mode is needed (plan §3.1 shows
`preprocess-book input.pdf --inspect`), introduce it as a sibling
subcommand `inspect` rather than a flag (per A-1.md §3.4's mapping table).

### 4.3 Page separator: form feed (`b"\x0c"`)

Pages in the extracted `.txt` output are separated by a form-feed byte.
This matches the PyMuPDF docs convention and makes the file trivially
splittable later (`text.split(b"\x0c")`). Exposed as
`extract.PAGE_SEPARATOR` for any future consumer.

### 4.4 Subcommand dispatch pattern

The pattern future subcommand modules MUST follow:

```python
# my_subcommand.py

SUBCOMMAND = "my-subcommand"

def add_subparser(subparsers: argparse._SubParsersAction) -> argparse.ArgumentParser:
    parser = subparsers.add_parser(SUBCOMMAND, help="...", description="...")
    # ... add_argument calls ...
    parser.set_defaults(func=run)
    return parser

def run(args: argparse.Namespace) -> int:
    # ... do work ...
    return 0  # or non-zero exit code
```

`cli.main()` dispatches via `getattr(args, "func", None)`. This is the
stdlib-idiomatic argparse subcommand pattern; do not introduce a custom
registry or decorator-based system without a deliberate decision.

### 4.5 Empty-page detection: warn, do not error

`extract_text` counts pages whose `get_text()` returns only whitespace
and logs a single warning summary at the end if any exist. It does *not*
raise. This deliberately accommodates scanned-without-OCR PDFs: the user
gets a recognizable signal in the logs without the whole extraction
failing. OCR support is out of scope; it is not in any current ticket.

### 4.6 Exit code conventions

Established in `preprocess.run()`:
- `0` — success
- `2` — input file not found (`FileNotFoundError`)
- `1` — any other failure (with `logger.exception` for the traceback)

Future subcommand handlers should follow this. The `2` for "input not found"
matches Unix convention for "no such file or directory" use of exit codes.

### 4.7 Cumulative state docs convention (`PROJECT_API.md` / `PROJECT_TREE.md`)

A-2 establishes a new convention layered on top of A-1's per-ticket
synopses: two cumulative state docs at the repo root.

- **`PROJECT_API.md`** — cumulative API reference. Every module, every
  public symbol, every CLI subcommand, every dependency. Updated at the
  close of every ticket. Read at the start of every ticket.
- **`PROJECT_TREE.md`** — current directory layout with per-ticket
  `[NEW]` / `[MODIFIED]` / `[UNCHANGED]` annotations. Tags reset at the
  start of each new ticket.

These exist because per-ticket narrative synopses (like A-1.md and this
file) don't answer "what's the current signature of `extract_text`?" or
"does `preprocess.py` exist yet?" quickly enough. The cumulative docs do.

**The two doc types serve different purposes; keep both.** Synopses
record *why* decisions were made and what was learned. State docs record
*what currently exists*. Future agents read both, in that order.

### 4.8 PyMuPDF API surface used (verified)

A-2 uses these PyMuPDF entry points:
- `pymupdf.open(path)` — open a document
- `doc.page_count` — int attribute
- iteration: `for page in doc:`
- `page.get_text()` — returns str, default `"text"` mode
- `doc.close()` — explicit close

All of the above were user-verified during the A-2 session. The import
name is `pymupdf` (not `fitz`, which is the legacy import).

## 8. Hand-Off Notes for A-3

A-3 is "Chapter detection from outline" — parse a PDF's bookmark tree
into a chapter list with start pages. Acceptance: tested PDF with
bookmarks produces a chapter list matching the table of contents.

Things the next agent should know before starting A-3:

1. **Read `PROJECT_API.md` and `PROJECT_TREE.md` first.** They reflect
   the current state of the project; this synopsis is the narrative.

2. **Verify A-2 was actually applied to the repo** before starting A-3.
   If `extract.py`, `preprocess.py`, and the `cli.py` modifications are
   missing, A-2 was drafted but never applied. Confirm with the user.

3. **Create a new module, don't bolt onto `preprocess.py`.** Per §4.1,
   library logic goes in its own module. Suggested name: `chapters.py`
   under `src/gbc_reader_prep/`. Public surface might be something like:
   ```python
   def detect_chapters_from_outline(doc: pymupdf.Document) -> list[Chapter]:
       ...
   ```
   where `Chapter` is a `dataclass` with at minimum `title: str`,
   `start_page: int`.

4. **`preprocess.run` should optionally surface chapter info** for
   debugging/visibility. A `--show-chapters` flag (or a separate
   `inspect` subcommand, per plan §3.1) is a reasonable UX choice for
   A-3's acceptance check. A-1.md §3.4 maps `--inspect` from the plan to
   an `inspect` subcommand; A-5 is where the trimming dry-run lands, so
   it may be cleaner to wait until then to introduce the subcommand.
   **Open question for the user when A-3 starts.**

5. **PyMuPDF outline API:** `doc.get_toc()` returns a list of
   `[level, title, page]` entries (1-indexed pages). I am **not** 100%
   confident in that signature — verify against the PyMuPDF docs at the
   time of A-3 (it's been stable historically but always check).

6. **What if the PDF has no outline?** A-3 acceptance is for PDFs *with*
   bookmarks. A-4 covers the no-bookmark heuristic fallback. So in A-3,
   if `doc.get_toc()` returns an empty list, log a warning and continue
   without crashing — the heuristic in A-4 is the proper handler.

7. **Test fixture strategy:** to actually exercise outline parsing in a
   unit test, you can generate a small PDF with a known outline using
   PyMuPDF itself (`doc.set_toc(...)`). This avoids needing a real
   external PDF for CI. Worth setting up for A-3 since the project will
   need PDF fixtures from here on.
