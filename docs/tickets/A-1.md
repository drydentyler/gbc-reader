## 3. Architectural Decisions Made in A-1

These choices were made during A-1 and should be respected by later tickets
unless deliberately changed (and documented).

### 3.1 `src/` layout
Package code lives in `src/gbc_reader_prep/` rather than at the repo root.
This is modern Python best practice — it prevents accidentally importing the
in-development source instead of the installed version, and it forces you to
run from an installed package, which catches packaging mistakes early.

### 3.2 Build backend: hatchling
`pyproject.toml` declares `hatchling` as the build backend. The version
string is read **dynamically** from `src/gbc_reader_prep/__init__.py` via:

```toml
[tool.hatch.version]
path = "src/gbc_reader_prep/__init__.py"
```

**To bump the version in any future ticket, edit only `__init__.py`.** Do
not put a literal `version = "..."` line back into `[project]` — that would
conflict with `dynamic = ["version"]`.

### 3.3 CLI library: argparse (stdlib)
Used argparse instead of click. The project plan §3.1 mentioned either was
acceptable; argparse was chosen to avoid an external dependency for a CLI
that's currently very small. Future subcommands should extend the existing
parser via `parser.add_subparsers()` — do not switch to click without a
deliberate decision and migration of the existing `--version` / `--verbose`
flags.

### 3.4 CLI command name: `gbc-reader-prep` (not `preprocess-book`)
**This resolves a conflict in the project plan.** Plan §3.1 shows the CLI
invoked as `preprocess-book`, but ticket A-1's acceptance criteria require
`gbc-reader-prep`. We followed A-1.

**Resolved direction confirmed by the user**: future tickets will use a
**subcommand** structure under the single `gbc-reader-prep` script. Mental
translation for plan §3.1 examples:

| Plan §3.1 example                            | Actual command from A-2 onward                          |
| -------------------------------------------- | ------------------------------------------------------- |
| `preprocess-book input.pdf --output out.book`| `gbc-reader-prep preprocess input.pdf --output out.book`|
| `preprocess-book input.pdf --inspect`        | `gbc-reader-prep inspect input.pdf`                     |
| `preprocess-book input.pdf --skip-trim`      | `gbc-reader-prep preprocess input.pdf --skip-trim`      |

### 3.5 Testing: pytest
- Configured via `[tool.pytest.ini_options]` in `pyproject.toml`
  (`testpaths = ["tests"]`, `addopts = "-ra"`).
- Run with `pytest` from the project root (venv activated).
- Tests are kept under `tests/` mirroring the source structure.
- A-1 includes only smoke tests covering the acceptance criteria; later
  tickets should add unit tests in the same directory as new modules are
  added.

### 3.6 Python version: `>=3.11`
Set in `pyproject.toml` per project plan §3.1. This is a minimum, not a
target — newer Pythons satisfy it. Do not bump the floor without a reason
that's worth the compatibility cost.

### 3.7 Logging convention
- Module-level logger named `gbc_reader_prep` is initialized at the top of
  `cli.py`.
- Format: `%(asctime)s %(levelname)-7s %(name)s: %(message)s`,
  datefmt `%Y-%m-%d %H:%M:%S`.
- INFO by default; DEBUG when `--verbose` is passed.
- **Future modules should obtain loggers as
  `logging.getLogger("gbc_reader_prep.<submodule>")`** so all project logs
  share a common root and can be filtered together.

## 6. Development Environment Notes

The user is on **Windows** using **PyCharm**. Recurring gotchas observed
during A-1 setup that future agents should anticipate:

- **Use `py` not `python`** outside an activated venv on Windows. The
  Microsoft Store stub at `WindowsApps\python.exe` intercepts the `python`
  command if the real Python wasn't added to PATH during install. The `py`
  launcher is on PATH unconditionally.
- **Activate venv with** `.venv\Scripts\Activate.ps1` in PowerShell. Once
  activated, `python` correctly resolves to the venv's interpreter.
- **PowerShell execution policy**: first-time users may hit "running scripts
  is disabled on this system" when activating. Fix:
  `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`.
- **PyCharm Sources Root**: with the `src/` layout, PyCharm may not auto-
  detect the package location. If the user reports red squiggles on imports,
  have them right-click the `src/` folder → **Mark Directory as → Sources
  Root**.
- **Verifying you're in the right Python**: `pip --version` prints the
  Python path. Useful for confirming the venv is active before installing
  anything.

## 8. Specific Hand-Off Notes for A-2

A-2 is "PDF text extraction proof-of-concept" — extract text from a sample
PDF with PyMuPDF and save to `.txt` for visual inspection across 3 PDFs
(novel, technical book, illustrated book).

Things the next agent should know before starting A-2:

1. **Add `pymupdf` to `[project].dependencies`** in `pyproject.toml`. The
   import name is `import pymupdf` (modern; some older code uses
   `import fitz`). Verify the current recommended import on the PyMuPDF
   project page before committing.
2. **Project plan open question Q6** ("verify MuPDF/PyMuPDF licensing
   satisfies your intent") should be raised with the user **before** the
   pymupdf dependency is added. MuPDF is dual-licensed (AGPL + commercial).
   For personal/private use it's fine, but the user should explicitly
   acknowledge it.
3. **The 3 sample PDFs** need to be sourced. Ask the user to supply them,
   or use clearly-licensed test PDFs (e.g., Project Gutenberg for a novel).
   Do not invent or assume PDF content.
4. **A-2's output is a debug artifact** — a plain `.txt` for visual
   inspection. It is *not* the `.book` file format. The `.book` format
   doesn't get written until A-8.
5. **Subcommand to add**: per §3.4 above, add the `preprocess` subcommand
   skeleton in A-2 (even if it only writes `.txt` so far). Wire it through
   `parser.add_subparsers()` in `cli.py`.
6. **First module to add**: probably something like
   `src/gbc_reader_prep/pdf_io.py` for PDF reading. Get a logger via
   `logging.getLogger("gbc_reader_prep.pdf_io")`.
