## 1. What Was Delivered

A-7 adds the pagination engine: given per-page extracted PDF text and a
chapter list, lay out text into 400x240 display pages against a
fixed-width character grid, enforcing the rule that a chapter's first
page always begins with that chapter's first line at the top of the
display.

**Source files:**

- `src/gbc_reader_prep/paginate.py` (new) —
  - `FontMetrics` — frozen dataclass: `char_width_px`, `line_height_px`.
  - `DEFAULT_FONT_METRICS` — placeholder 6x10px grid (no real firmware
    font exists yet; see "Design Notes" below).
  - `load_font_metrics(path)` — loads a `FontMetrics` from a JSON file.
  - `chars_per_line(font_metrics, display_width)` /
    `lines_per_page(font_metrics, display_height)` — grid arithmetic.
  - `wrap_text(text, max_chars)` — greedy word-wrap with whitespace
    collapsing and hard-splitting of over-long words.
  - `Page` — frozen dataclass: `chapter_id`, `lines` (always padded to a
    fixed length).
  - `paginate_chapters(page_texts, chapters, font_metrics, ..., start_page, end_page)`
    — the core layout function.
  - `paginate_path(pdf_path, ...)` — path-based convenience wrapper tying
    together chapter detection, text extraction, and trim-bounds
    auto-detection.
- `src/gbc_reader_prep/extract.py` (modified) —
  - `extract_text_pages(pdf_path) -> list[str]` — in-memory per-page text
    extraction, used by the pagination engine instead of re-parsing the
    form-feed-delimited `.txt` output.
- `src/gbc_reader_prep/preprocess.py` (modified) —
  - `--paginate` flag: lays out the detected main-content text and logs a
    page-count summary (overall + per-chapter). Works with or without
    `--inspect`.
  - `--font-metrics` flag: path to a JSON font metrics file overriding
    `DEFAULT_FONT_METRICS`.
  - `_load_font_metrics(args)` and `_report_pagination(...)` private
    helpers, called from both `run()` and `_report_trim()`.

**Tests:**

- `tests/test_paginate.py` (new) — 19 tests: font-metrics arithmetic and
  loading, word-wrap edge cases, chapter-start-at-top enforcement,
  page-range bounds, the ~250-words/page sanity check from the
  acceptance criterion, end-to-end `paginate_path`, and CLI integration
  for `--paginate`/`--font-metrics` (including the missing-file error
  path).

**Acceptance status (per project plan §4 / issue #6):**

  `test_paginate_chapters_sanity_word_count` (50,000 words → 100-400

- "Chapter boundaries are respected — each chapter starts at the top of a

## 2. Design Notes

- **No real font metrics exist yet.** B-4 (firmware: pick/generate a
  bitmap font) hasn't landed, so there's no glyph data to lay out
  against. `paginate.py` works against an abstract fixed-width character
  grid (`FontMetrics`) instead of real glyphs. `DEFAULT_FONT_METRICS`
  (6x10px) is a placeholder chosen only to land in the acceptance
  criterion's word-count ballpark — **it must be replaced** with the
  real firmware font's actual cell dimensions once B-4 picks one (see
  project plan §6 Q7, which explicitly blocks this exact alignment
  between A-7 and B-4). Until then, `--font-metrics` lets a caller
  override the grid with real numbers as soon as they're known, without
  changing any pagination logic.
- **Character-grid model, not proportional layout.** The firmware's
  Sharp memory LCD + `Adafruit_GFX` rendering path (per the project
  plan's `renderPage` sketch) implies the eventual font is a simple
  fixed-width bitmap font, so a fixed-width grid model on the desktop
  side is the right level of fidelity — adding proportional-width text
  measurement now would model detail the firmware doesn't have anyway.
- **Chapter-start-at-top via buffer flush, not post-hoc fixup.** Rather
  than laying out all chapters' text as one continuous stream and then
  scanning for chapter boundaries to insert padding, `paginate_chapters`
  processes chapters one at a time and flushes (blank-padding) any
  leftover buffer from the previous chapter before starting the next.
  This makes the invariant structural — there's no way for a chapter's
  first line to land anywhere but a fresh page — at the cost of
  generally one extra padded page per chapter (an explicit tradeoff
  given the project plan's own wording: "pad the previous page with
  blank lines if needed").
- **No-chapters fallback.** If no chapter falls inside the requested page
  range (e.g. a PDF with no outline and no heuristic matches),
  `paginate_chapters` treats the whole range as a single unnamed chapter
  rather than producing zero pages or raising — matches the rest of the
  codebase's "degrade gracefully, never lose the book's content" stance
  (see `chapters.py`/`trim.py`).
- **`extract_text_pages` added rather than reusing `extract_text`'s file
  output.** The pagination engine needs per-page text as Python strings,
  not a form-feed-delimited file on disk; round-tripping through a
  written `.txt` file and re-splitting it would be a strictly worse path
  for an in-process caller, so `extract.py` gained a small in-memory
  sibling function instead.

## 3. Follow-ups for Later Tickets

- A-8 (`.book` file writer) should call `paginate_path` (or
  `paginate_chapters` directly, if it already has chapters/page-texts/
  bounds from its own pipeline) and place the resulting `Page` records
  into the `.book` schema's `pages` array (`chapter_id` → `chapter_id`,
  `lines` → `lines`), and the chapter list's `id` field should match
  `Page.chapter_id`'s indexing (0-based, in chapter order within the
  trimmed range) for the two to line up correctly.
- **B-4 must supply real font metrics.** Once the firmware's bitmap font
  is chosen, compute its actual `char_width_px`/`line_height_px` and
  either update `DEFAULT_FONT_METRICS` or pass them via `--font-metrics`
  for every real preprocessing run — using the 6x10px placeholder against
  a real book would silently produce display pages that don't match what
  the firmware actually renders.
- A-9 (end-to-end CLI test) should run `--paginate` against all 3 real
  test books and eyeball the per-chapter page counts for obvious
  pathologies (e.g. a chapter producing 1 page when it's clearly 20 pages
  of text, which would indicate a wrap-width or bounds bug not caught by
  the synthetic fixtures here).

## 4. Follow-up: Manual Review Output, Chapter Title Pages, and Fit Verification

Three additional small enhancements landed after the initial A-7 delivery,
in response to manual testing feedback:

- **`--paginate-output <path>`** (`preprocess.py`): dumps the full
  laid-out page contents to a `.txt` file — one fixed-width block per
  display page (lines exactly as they'll render, blank padding included),
  with a header naming the 1-based page number and chapter — so a human
  can actually eyeball where page and chapter breaks fall, rather than
  trusting only the page-count summary `--paginate` logs. Works with or
  without `--inspect`. Added `_write_paginate_output` as a private helper
  in `preprocess.py`; note it must rebuild the same in-range,
  `start_page`-sorted chapter subset (with the single-fallback-chapter
  rule) that `paginate_chapters` uses internally to assign `chapter_id`,
  since that's what the IDs in `Page.chapter_id` actually index into.
- **Chapter title pages** (`paginate.py`): each chapter with a non-blank
  title now gets a dedicated title page — blank except for the chapter's
  title, centered both horizontally and vertically — inserted immediately
  before that chapter's first body page. Implemented via two new pure
  functions, `center_line` (horizontal centering with spaces, truncating
  if the text is too wide) and `make_title_page` (word-wraps an
  over-long title, vertically centers the resulting line block, and
  horizontally centers each line), plus a `Page.is_title_page: bool`
  field (default `False`) so callers — including `_write_paginate_output`,
  which now tags these `[TITLE PAGE]` in its dump — can distinguish title
  pages from body pages. The synthetic single-chapter fallback used when
  no chapters are detected has a blank title and so gets no title page,
  consistent with A-7's existing "don't decorate text we don't actually
  know belongs to a named chapter" stance. This does mean each named
  chapter now costs one extra page versus the original A-7 page counts —
  an intentional tradeoff for a clear, unambiguous chapter boundary on
  the device, mirroring the existing chapter-start-at-top tradeoff
  already documented above.
- **"Does this fit the 400x240 display?" verification**: no code change
  was needed here — `chars_per_line`/`lines_per_page` are floor-divided
  from `display_width`/`display_height`, so every wrapped line and every
  page is mathematically guaranteed to fit within the configured display
  dimensions, regardless of how much source text there is. The only real
  risk is supplying `--font-metrics` values that don't match the
  eventual real firmware font — already a known, documented B-4
  dependency (see §2 above) — not a pagination-engine correctness issue.

Tests: `tests/test_paginate.py` grew from 19 to 28 tests (new:
`center_line` centering/truncation, `make_title_page` horizontal+vertical

start/end-page-bounds tests now account for the extra title page per
named chapter; CLI `--paginate-output` dump contents, with and without
`--inspect`). Full suite: 79/79 passing.
